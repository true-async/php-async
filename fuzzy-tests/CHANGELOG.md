# Fuzzy-tests changelog

Test-coverage and harness changes for `ext/async/fuzzy-tests/`. Kept
separate from the main `CHANGELOG.md` so user-facing extension features
are not mixed with test-suite history.

## Unreleased

### Added
- **#143 IO chaos — ext/sockets POSIX socket_* under cancel** — `io/socket_ext_chaos.feature` (4 scenarios → 6 .phpt). First chaos coverage of the ext/sockets API path (`xp_socket.c` / `network_async.c`), distinct from the streams layer covered by `io/cancel_during_connect`, `udp_chaos`, `cancel_during_io`. Three verbs exercised on the cancel surface: `socket_connect` to a TCP blackhole (RFC 5737 TEST-NET-1, cancel-timing sweep 0/5/50 ms), `socket_recvfrom` on a fresh bound UDP socket (no datagrams ever arrive, cancel after 20 ms), `socket_accept` on a fresh TCP listener (no clients ever connect, cancel after 20 ms). New steps inline the socket lifecycle per-step (no shared registry needed for one-shot ops). New SKIPIF tag `sockets`. Confirmed via separate probe that ext/sockets is reactor-integrated (recvfrom parks ~50 ms before cancellation lands). 6 .phpt × 4 seeds green plus ASAN-ZTS — no findings.
- **#143 IO chaos — SSL accept-side under concurrent handshakes + cancel** — `io/ssl_accept_chaos.feature` (4 scenarios → 7 .phpt). `tests/stream/027-ssl_concurrent_accept.phpt` pins the deterministic "three SSL servers each accept one client" shape (regression backstop for `network_async_accept_incoming()` event-loop conflicts); the existing `tls_connect.feature` (#138) covers client-side cancellation. This feature is the accept-side mirror: three independent ssl:// servers each accept one client under random scheduling, one server fans in four concurrent TLS clients, and the acceptor itself is cancelled while parked in `stream_socket_accept()` (timing sweep across 0/5/50/200 ms). Reuses existing `TLS server "S" listening on "SRV" accepting up to N clients` + TLS client step. 7 .phpt × 5 seeds green plus ASAN-ZTS — no findings, the accept path is solid.
- **#143 IO chaos — flock() under N-coroutine contention** — `io/flock_chaos.feature` (2 scenarios → 2 .phpt). `tests/io/081-flock_non_blocking_event_loop.phpt` pins one deterministic shape; this feature fuzzes the contention surface. New `Context::$lockFiles` registry + steps `Given a shared lock file "L"`, `coroutine "X" acquires LOCK_EX on "L" then releases after N ms` with counter family `flock_{attempts,ok,cancelled,failed}`. Scenarios: four coroutines contend, holder + two waiters all eventually acquire. 2 .phpt × 3 seeds green plus ASAN-ZTS. The three planned cancel-mid-flock scenarios are commented under `# Blocked: #146`: drafting them surfaced **a real stack-use-after-return** in `main/streams/plain_wrapper.c:1192` — the flock task struct lives on the caller's stack but the libuv worker thread keeps writing to it after the coroutine unwinds on cancel. Confirmed by ASAN trace; filed as #146.
- **#143 IO chaos — feof() semantics under concurrent close / drain / poll** — `io/feof_chaos.feature` (4 scenarios → 6 .phpt). `tests/stream/038–044` pin the deterministic feof() return value at specific points (live socket, after recv, after remote close, with pending data). This feature crosses those into chaos: a writer pushes N bytes then closes the write end, a reader drains via canonical `while (!feof) fread` loop, an optional third coroutine spams `feof()` from a sibling context. Invariants: drain loop receives every byte BEFORE feof becomes true (no premature-EOF data loss), drain/cancel/fail outcomes sum to attempts, poller true/false outcomes sum to attempts. New steps `coroutine "X" drains shared pipe "P" with feof loop` (counter family `feof_drain_*` incl. `feof_drain_bytes`) and `coroutine "X" polls feof of shared pipe "P" N times` (`feof_poll_*`). Scenarios: drain-with-close, close-timing sweep (0/5/25 ms), concurrent poller, cancel mid-drain. 6 .phpt × 5 seeds green plus ASAN-ZTS.
- **#143 Layer 3 HTTP chaos — async ext/curl curl_multi** — `curl/curl_multi_chaos.feature` (3 scenarios → 3 .phpt). First chaos coverage of `ext/curl/curl_async.c`'s `curl_multi_select()` reactor integration (single-handle `curl_exec` already covered by #136 `http_chaos.feature`). One coroutine attaches N easy handles to a `curl_multi_init()`, drives the standard `curl_multi_exec` / `curl_multi_select` loop, drains `CURLMSG_DONE` into per-handle outcome buckets. New step `coroutine "X" fetches peers "EP1","EP2",… via curl_multi` + `StandardSteps::curlMulti()`. Scenarios: three peers all return intact, one peer aborts mid-body while the others complete, two coroutines each owning their own multi handle (stresses the per-multi watcher bookkeeping). 3 .phpt × 4 seeds green, plus ASAN-ZTS. The three planned cancel-mid-curl_multi_select scenarios are commented in the feature under `# Blocked: #145`: drafting them surfaced **a real heap-corruption bug** — `AsyncCancellation` interrupting `curl_multi_select()` corrupts the zend allocator's free-list and the next coroutine SEGVs in `zend_mm_alloc_small`. Reproduced in ~50 lines outside the harness on ASAN-ZTS; filed as #145.
- **#143 Layer 3 process chaos — proc_open / proc_close storm** — `exec/proc_open_chaos.feature` (1 scenario → 1 .phpt). First chaos coverage of `proc_open()`/`proc_close()` outside `ext/async/tests/exec/`. Four coroutines each cycle proc_open + proc_close 5 times against the test PHP binary (`exit(0);`), yielding via `Async\suspend()` per iteration so the reactor's `process_events` hashtable is hammered under random scheduling. Backstops `tests/exec/011-proc_open_handle_reuse_uaf` (deterministic, sequential): this fuzzes the same code path with concurrent open/close cycles. New `Context::$processes` registry + steps `Given a long-lived child process "C" sleeping N ms`, `coroutine "X" reads stdout of child process "C"`, `coroutine "X" closes child process "C"`, `coroutine "X" sends SIGTERM to child process "C"`, `coroutine "X" runs proc_open + proc_close N times`. New SKIPIF tag `proc-open` (POSIX-only, requires `TEST_PHP_EXECUTABLE`). The parked-fread-vs-close scenarios are commented in the feature under `# Blocked: #144`: while drafting them the chaos repro surfaced a **real reactor bug** — `proc_terminate` + `proc_close` does not wake a parked `fread()` on the child's stdout pipe, and the deadlock detector eventually aborts. Reproduced outside the harness in ~25 lines; tracked separately as #144. Storm scenario green under fifo + 4 random seeds, plus ASAN-ZTS.
- **#138 DB chaos — pool + reset_peer mixed workload** — `db/pool_max_reset_chaos.feature` (2 scenarios → 4 .phpt). Bounded MySQL pool running a mixed transaction + query workload while a Toxiproxy `reset_peer` toxic kills the connection mid-flight. Cross-product of existing `mysql_chaos.feature` (`reset_peer` over a single work type) with the mixed-workload pattern. Pool must surface every RST cleanly, reclaim slots, leave no orphans. Two scenarios: four coroutines (2 txn + 2 query) on a pool=4 with reset at 30 ms; reset-timing sweep across 5/60/200 ms. **Surfaced a real behavioural gap** while designing the feature: over-saturated pool (N coroutines > pool_size) + permanent `reset_peer` deadlocks pool waiters — every new connect attempt is RST'd, so a queued waiter never acquires a slot. Documented in the feature header as out-of-scope (tracked separately). 4 .phpt × 8 seeds green.
- **#138 DB chaos — PDO pool stmt cache LRU** — `db/stmt_cache_chaos.feature` (4 scenarios → 7 .phpt). First chaos coverage of `PDO::ATTR_POOL_STMT_CACHE_SIZE`. Drives cache fill + LRU eviction by preparing many distinct SQL strings against a small-capacity cache; covers concurrent storms on a shared pool and cancel mid-storm. SQLite is used so the test runs without Toxiproxy/MySQL — the cache itself lives in `ext/pdo/pdo_pool.c` and is driver-agnostic. New `ChaosNet::setEvilDbStmtCache()` + step `Given a pooled SQLite database "DB" with N connections and stmt cache C`, and `coroutine "X" runs cache-storm of N statements on database "DB"` with `db_storm_{attempts,ok,cancelled,failed,rows,no_db}` counter family. Cache-storm step yields via `\Async\delay(1)` per iteration so cancellation lands on SQLite's synchronous prepare path. Scenarios: storm > capacity, four coroutines sharing one pool, cancel mid-storm, cache-capacity sweep (undersized → equal → oversized). 7 .phpt × 8 seeds green.
- **#138 Layer 1 IO chaos — async DNS resolve** — `io/dns_slow.feature` (4 scenarios → 7 .phpt). First chaos coverage of `php_network_getaddrinfo_async()` (libuv DNS worker thread). `stream_socket_client('tcp://*.invalid:80')` always NXDOMAINs after a ~30 ms worker-thread roundtrip on glibc, giving a cancel window. New step `coroutine "X" resolves nonexistent hostname "H" with timeout N ms` with `dns_{attempts,ok,failed,cancelled,timeout}` outcome family + SKIPIF tag `dns-async-engages` (sync probe must take ≥20 ms; musl / fully-cached NSS hosts that collapse NXDOMAIN to microseconds are skipped to avoid faking coverage). Scenarios: smoke NXDOMAIN, cancel mid-resolve, cancel-timing sweep across the resolve window, 4 concurrent resolves. 7 .phpt × 8 seeds green.
- **#138 Layer 1 IO chaos — FileSystemWatcher** — `io/fs_watcher_chaos.feature` (4 scenarios → 4 .phpt). First chaos coverage of `Async\FileSystemWatcher` outside `ext/async/tests/fs_watcher/`. Iterator coroutine `foreach`es the watcher; producer coroutines `file_put_contents` into the watched dir on staggered delays; a third coroutine either `close()`s the watcher or cancels the iterator. New `Context::$fsWatchers` registry with tempdir lifecycle (close in run() teardown, rmdir in __destruct). New steps `Given a filesystem watcher "W" on a fresh temp directory`, `coroutine "X" iterates filesystem watcher "W"`, `coroutine "X" touches file "F" in filesystem watcher "W" directory`, `coroutine "X" closes filesystem watcher "W"` with the `fsw_{iter_attempts,events,iter_done,iter_cancelled,iter_failed,bad_event,touch_*}` families. Scenarios: single touch + close, three producers race close, cancel iterator mid-foreach, immediate close with no events. 4 .phpt × 10 seeds green.
- **#138 Layer 1 IO chaos — UDP recvfrom** — `io/udp_chaos.feature` (4 scenarios → 6 .phpt). Scenarios: receive single payload, cancel mid-recv, two senders / one receiver, send-vs-cancel timing sweep. 6 .phpt × 8 seeds green.
- **#138 Layer 1 IO chaos — stream_select** — `io/stream_select_chaos.feature` (5 scenarios → 8 .phpt). Drives the reactor's `network_async_stream_select()` (multi-fd poll watcher) by calling `stream_select()` over a set of shared pipes; another coroutine writes to one mid-wait, a killer may cancel the selector. - **#138 Layer 1 IO chaos — TLS handshake** — `io/tls_connect.feature` (3 scenarios → 6 .phpt). `stream_socket_client('ssl://…')` drives TCP connect + TLS handshake; both phases yield to the reactor and cancel/timeout must release the connect-watcher AND any in-flight crypto-poll state without leaks. New `Context::$tlsServers` registry + in-process echo server (ssl listener using the existing `ext/async/tests/stream/ssl_test_cert.pem` cert + key, accept N, send "ok", close). Steps `TLS server "S" listening on "SRV" accepting up to N clients` and `coroutine "X" connects via TLS to server "S" with timeout N ms` with the `io_tls_{attempts,ok,timeout,cancelled,failed}` outcome family + matching `tls_accept_*` server-side counters. SKIPIF tag `openssl`. Scenarios: smoke handshake, 4 concurrent clients (multiple in-flight crypto states), cancel-timing sweep across the handshake window. 6 .phpt × 9 seeds green.
- **#138 signal chaos — Async\signal() race coverage** — `signal/signal_chaos.feature` (5 scenarios → 7 .phpt). Backstop for php-async #109 (multi-thread race in `zend_signal_activate`, fixed via the `ZEND_ASYNC_REACTOR_IS_ENABLED` skip): exercises `Async\signal()` with concurrent raises (`posix_kill(getmypid(), SIGUSR{1,2})`), multiple waiters on the same signal, two signal kinds in parallel, and killer-cancels-waiter races against the raise. New steps `coroutine "X" awaits signal SIGUSR{1,2}` and `coroutine "X" raises signal SIGUSR{1,2}`, SKIPIF tag `signal` (Windows + posix). `Runner` now installs `pcntl_signal(SIGUSR{1,2}, SIG_IGN)` as a safety net at scenario startup — Async\signal() registration overrides SIG_IGN while active; the IGN catches stray raises after a waiter was cancelled (otherwise the unhandled signal default-terminates the test process). 7 .phpt × 8 seeds green.
- **#138 Layer 1 IO chaos — concurrent file writes** — `io/file_concurrent_writes.feature` (4 scenarios → 6 .phpt). Regression backstop for `php_stdiop_write` (#129/#133): every coroutine writing the SAME file descriptor parks on its single shared async-IO event; a write completing for any coroutine wakes them all, and a spuriously-woken coroutine must re-suspend until its own request completed — before the fix it disposed the in-flight libuv request and writes were silently lost on macOS/Windows. `tests/io/083` only pins a deterministic 4-worker scenario; this feature crosses worker count, chunk size, iteration count, and interleaved sleeps under random ChaosNet scheduling. New `Context::$files` registry + steps `Given a shared file "F"`, `coroutine "X" writes N chunks of M bytes to shared file "F"`, `coroutine "X" closes shared file "F"`, and variadic invariant `shared file "F" byte size equals counter "A" plus counter "B" ...` — the kernel-observed file size must equal the sum of bytes every coroutine claims fwrite() accepted, exactly the assertion the spurious-wakeup bug broke. Tempfile lifecycle: handle flush+close in run() teardown so Then assertions can filesize() it, path unlink deferred to `Context::__destruct()`. 6 generated .phpt × 10 seeds green.
- **#138 Layer 1 IO chaos — IPv6 connect-watcher** — `io/connect_v6.feature` (3 scenarios → 3 .phpt). Same race surface as the v4 features but on an AF_INET6 socket; uses `[::ffff:192.0.2.1]:81` (v4-mapped TEST-NET-1) so the v6 socket-creation branch is exercised while the SYN still goes out to a routable address and the connect actually parks. Includes a v4+v6 mixed scenario to catch family-specific bugs in the shared `network_async_await_stream_socket()` release path. New step `coroutine "X" connects to IPv6 blackhole "ADDR" with timeout N ms` and SKIPIF tag `tcp-blackhole-v6` (skips hosts without v6 egress).
- **#138 Layer 1 IO chaos — connect-watcher** — `io/cancel_during_connect.feature` (3 scenarios → 8 .phpt) and `io/connect_with_timeout.feature` (3 scenarios → 5 .phpt). Exercises `network_async_await_stream_socket()` (the reactor's connect-watcher) via `stream_socket_client()` against an RFC 5737 TEST-NET-1 blackhole (`192.0.2.1:81`): SYN goes out, no response, kernel returns EINPROGRESS, the connect parks in poll. Killer cancel mid-wait and explicit short-timeout fire — both must release the watcher without UAF or libuv handle leak. New step `coroutine "X" connects to TCP blackhole "ADDR" with timeout N ms` with the `io_connect_{attempts,ok,timeout,cancelled,failed}` outcome family, plus new generic step `counter "X" is at least N` for dominant-bucket assertions. New SKIPIF probe `tcp-blackhole`: sanity-checks that the chosen blackhole IP actually suspends past 50 ms on this host — Alpine/CI hosts without a default route would otherwise fail synchronously with ENETUNREACH and never engage the connect-watcher, silently lying about coverage. Connect cancel timeouts capped at 5 s (was 30 s) so a regression in the cancel path fails fast instead of hanging. Every cancel/timeout scenario now asserts `io_connect_{cancelled,timeout}` ≥ 1 — turns the sum-only invariant into a real check that the targeted code path actually fired. 13 generated .phpt × 8 seeds green (WSL2 dt≈50 ms for a 50 ms timeout confirms the watcher is engaged).
- **#138 Layer 1 IO chaos — close & concurrent-readers** — `io/stream_close_during_read.feature` and `io/concurrent_readers.feature` (5 scenarios incl. 4-reader cancel-half + delay-varied outline → 20 .phpt). Cross-coroutine `fclose()` of the write end races a parked `fread()`; N coroutines park on one shared `stream_socket_pair`; cancellation of half the readers must leave the survivors unaffected. New `Context::pipes` registry + steps: `Given a shared pipe "P"`, `coroutine "X" reads from shared pipe "P"`, `coroutine "X" writes "<msg>" to shared pipe "P"`, `coroutine "X" closes the write end of shared pipe "P"`. **Found a real bug** — cancelling one of N readers on a shared poll proxy double-stopped the proxy and parked the survivors forever; fixed in php-src `Zend/zend_async_API.{c,h}` and `ext/async/scheduler.c` (ABI bumped to v0.18.0). See `FINDINGS.md`.

- **#136 SQLite pool-focused coverage** — `db/sqlite_chaos.feature` (6 scenarios). Local file, no Toxiproxy, pdo_sqlite operations do not yield — exercises only the PDO connection pool itself (per-coroutine `sqlite3*` slots over one shared file). New steps `a [pooled] SQLite database "DB"`. Schema seeded per-scenario into a per-PID file under `sys_get_temp_dir()`, removed in tearDown. Only `pdo_sqlite` required by SKIPIF.
- **#136 Concurrent transactions and heterogeneous workloads** — `db/concurrent.feature` (9 scenarios → 22 .phpt) targets many coroutines doing different things at once on one pooled `$pdo`: concurrent transactions, writer racing reader (reader's `WHERE id<=5` snapshot is interleaving-independent of writer's `id>5` insert), mixed workloads, a transaction cancelled while a sibling reader keeps working, a storm on an undersized pool. `StandardSteps::dbTransaction` runs a multi-statement transaction now (BEGIN → INSERT → read-back SELECT → COMMIT) so each yields several times. **Found a UAF in the PDO connection pool** under coroutine cancellation mid-transaction — fixed in php-src `ext/pdo/pdo_pool.c`.
- **#136 PgSQL coverage** — `db/pgsql_chaos.feature` (12 scenarios → 30 .phpt) mirrors `mysql_chaos.feature` against a Toxiproxy-fronted PostgreSQL server. `ChaosNet::openDbConnection()` and `dbUpstream()` driver-aware (mysql / pgsql); slow-query step picks `SELECT pg_sleep(2)` vs `SELECT SLEEP(2)` at action-run time. New steps `a MySQLi database "DB"` and `coroutine "C" queries|runs a slow query via|runs a transaction via mysqli "DB"`. `ChaosNet::openMysqliConnection()` with `MYSQLI_REPORT_STRICT`. - **#136 PDO MySQL coverage** — `db/mysql_chaos.feature` (12 scenarios → 30 .phpt). Async PDO MySQL through Toxiproxy: latency, bandwidth, TCP slicer, `reset_peer` mid-query / mid-transaction.
